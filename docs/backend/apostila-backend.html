Arquitetura Hexagonal (Ports & Adapters)

O domínio (hexágono interno) não conhece o mundo externo. Toda comunicação passa por portas (interfaces) e adaptadores (implementações concretas). Isso permite trocar banco, broker ou framework web sem alterar lógica de negócio.

Portas são interfaces em application/port/ — Adaptadores são implementações concretas em adapter/

Estrutura de Pacotes

// Cada microserviço segue esta organização
order-service/
├── domain/                     // Hexágono interno — Java puro
│   ├── model/                  // Entidades JPA, Value Objects
│   ├── event/                  // Eventos de domínio (records)
│   └── exception/              // Exceções de domínio
├── application/                // Use Cases + Portas
│   ├── port/in/                // O que o app sabe fazer
│   ├── port/out/               // O que o app precisa
│   ├── usecase/                // Orquestração
│   ├── dto/                    // Records (request/response)
│   └── mapper/                 // Entity ↔ DTO
└── adapter/                    // Implementações concretas
    ├── in/web/                 // Controllers, Filters, Security
    └── out/                    // JPA, Kafka, Feign, Config

Princípios SOLID no Projeto

S
Single Responsibility

Controller (HTTP), Use Case (orquestração), Entity (regras de domínio), Repository (persistência) — cada classe com 1 razão para mudar.

O
Open/Closed

OrderEventPublisher (porta) pode ter novas implementações (Kafka, RabbitMQ, SNS) sem alterar o código que a usa.

L
Liskov Substitution

Qualquer implementação de OrderRepository (JPA, in-memory) funciona sem quebrar o contrato.

I
Interface Segregation

Portas separadas: OrderEventPublisher publica eventos, OrderRepository persiste pedidos.

D
Dependency Inversion

OrderApplicationService depende de abstrações (portas), não de KafkaOrderEventPublisher (concreto).

Decisões Arquiteturais (ADRs)

ADR-001: Comunicação

REST síncrono (OpenFeign) para queries; Kafka assíncrono para eventos. Consistência eventual para fluxos de eventos.

ADR-002: Database-per-Service

PostgreSQL dedicado por serviço. Autonomia e isolamento. Sem JOINs cross-service; dados via eventos.

ADR-003: Flyway Migrations

SQL versionado (V1__, V2__). Rastreabilidade e reprodutibilidade. ddl-auto: validate.

ADR-004: JWT Stateless

Token self-contained; qualquer instância valida sem banco central. TTL fixo, sem revogação individual.

ADR-005: Hexagonal + DDD

Domínio no centro; portas e adaptadores. Entidades JPA no domain/ por pragmatismo. Mais adotado em fintechs.

💼 Perguntas de Entrevista — Fase 00

Explique comunicação síncrona vs assíncrona em microserviços.
Síncrona (REST/Feign): o chamador bloqueia e espera — bom para queries que precisam de resultado imediato. Assíncrona (Kafka): publica evento e segue — bom para reações desacopladas. Trade-off: síncrona dá acoplamento temporal; assíncrona dá eventual consistency.
O que é Arquitetura Hexagonal e por que usar?
Ports & Adapters: domínio no centro, comunicação via portas (interfaces) e adaptadores (implementações). Benefício principal: testabilidade — domínio testável sem Spring/banco/Kafka. Benefício secundário: substituibilidade — trocar JPA por MongoDB exige apenas novo adaptador.
Por que Database-per-Service e não banco compartilhado?
Autonomia (deploy independente), isolamento (migração de schema sem impactar outros), encapsulamento (dados acessíveis apenas via API). Trade-off: perda de transações distribuídas — use Saga ou Outbox.
Cite os 5 princípios SOLID com exemplos reais do projeto.
S: OrderApplicationService coordena apenas pedidos. O: OrderEventPublisher aceita KafkaPublisher ou RabbitPublisher. L: qualquer OrderRepository funciona sem quebrar contrato. I: portas separadas por responsabilidade. D: service depende de interfaces (portas), não de implementações.
Qual a diferença entre DDD e microserviços?
DDD é abordagem de modelagem (linguagem ubíqua, bounded contexts, entidades ricas). Microserviços é estilo arquitetural de deploy. DDD guia os limites de cada microserviço — cada bounded context tipicamente vira um serviço.

OrderStatus — Máquina de Estados

public enum OrderStatus {
    PENDING, CONFIRMED, PREPARING, READY,
    DELIVERING, DELIVERED, CANCELLED;

    public boolean canTransitionTo(OrderStatus target) {
        return switch (this) {
            case PENDING     -> target == CONFIRMED || target == CANCELLED;
            case CONFIRMED   -> target == PREPARING || target == CANCELLED;
            case PREPARING   -> target == READY;
            case READY       -> target == DELIVERING;
            case DELIVERING  -> target == DELIVERED;
            default         -> false;
        };
    }
}Java

Order — Aggregate Root

@Entity @Table(name = "orders")
public class Order extends BaseEntity {

    @Column(nullable = false)
    private Long customerId;

    @Column(nullable = false)
    private Long restaurantId;

    @Enumerated(EnumType.STRING)
    private OrderStatus status;

    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL, orphanRemoval = true)
    private List<OrderItem> items = new ArrayList<>();

    private BigDecimal totalAmount;

    // Factory method — único ponto de criação
    public static Order create(Long customerId, Long restaurantId) {
        Order order = new Order();
        order.customerId = customerId;
        order.restaurantId = restaurantId;
        order.status = OrderStatus.PENDING;
        order.totalAmount = BigDecimal.ZERO;
        return order;
    }

    // Regra de negócio vive na entidade (Rich Domain Model)
    public void addItem(OrderItem item) {
        items.add(item);
        item.setOrder(this);
        recalculateTotal();
    }

    public void transitionTo(OrderStatus newStatus) {
        if (!status.canTransitionTo(newStatus)) {
            throw new IllegalStateException(
                "Cannot transition from " + status + " to " + newStatus);
        }
        this.status = newStatus;
    }
}Java
💡 Factory Method vs Construtor Público

O Order.create() é um factory method: controla a criação, garante invariantes (status sempre PENDING, total zero) e torna a intenção explícita. Construtor público permitiria criar Order em estado inválido.

Camada de Aplicação — Use Case

@Service
@RequiredArgsConstructor
@Transactional(readOnly = true)
public class OrderApplicationService {

    private final OrderRepository orderRepository;
    private final OrderMapper mapper;

    @Transactional
    public OrderResponse createOrder(CreateOrderRequest request) {
        Order order = Order.create(request.customerId(), request.restaurantId());
        request.items().forEach(itemReq -> {
            OrderItem item = OrderItem.create(
                itemReq.menuItemId(), itemReq.menuItemName(),
                itemReq.quantity(), itemReq.unitPrice()
            );
            order.addItem(item);
        });
        return mapper.toResponse(orderRepository.save(order));
    }

    public Page<OrderResponse> findAll(Pageable pageable) {
        return orderRepository.findAll(pageable).map(mapper::toResponse);
    }
}Java

Exception Handler — ProblemDetail (RFC 7807)

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(OrderNotFoundException.class)
    public ProblemDetail handleNotFound(OrderNotFoundException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(
            HttpStatus.NOT_FOUND, ex.getMessage());
        pd.setTitle("Pedido não encontrado");
        return pd;
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ProblemDetail handleValidation(MethodArgumentNotValidException ex) {
        ProblemDetail pd = ProblemDetail.forStatusAndDetail(
            HttpStatus.BAD_REQUEST, "Erro de validação");
        pd.setProperty("errors", ex.getFieldErrors().stream()
            .map(fe -> fe.getField() + ": " + fe.getDefaultMessage())
            .toList());
        return pd;
    }
}Java

💼 Perguntas de Entrevista — Fase 01

Por que usar factory method em vez de construtores públicos?
Factory method controla criação, garante invariantes (status PENDING, total ZERO), torna intenção explícita e pode retornar subtipos. Construtor público permitiria criar entidade em estado inválido.
O que é um Aggregate Root no DDD?
É a entidade raiz de um agregado — o único ponto de entrada para modificar o cluster. Order é o Aggregate Root; OrderItem só é acessado via Order. Isso garante consistência das invariantes de negócio.
Qual a diferença entre DTO e Entity?
Entity tem identidade, ciclo de vida e regras de negócio (persiste no banco). DTO é objeto de transporte sem lógica — Java Record imutável que cruza fronteiras (web ↔ application). Nunca exponha entidades na API.
Por que usar ProblemDetail para erros REST?
RFC 7807 padroniza respostas de erro: type, title, status, detail, instance. Clientes parsam erros de forma uniforme. Spring Boot 3 suporta nativamente via ProblemDetail.

O Problema N+1 — Vilão de Performance JPA

N+1 transforma 1 query em 101 queries — JOIN FETCH resolve com um SQL JOIN

3 Soluções para N+1

CenárioSolução

Listagem com paginação@BatchSize(size = 50)

⚠️ Cuidado: JOIN FETCH + Paginação

JOIN FETCH com Pageable carrega TUDO em memória. Warning: HHH90003004: applying in memory! — Em 1M de pedidos, crashará a aplicação. Use @BatchSize para paginação.

PropagaçãoQuando usar

Chamar um método @Transactional de outro método da mesma classe NÃO ativa o proxy. O @Transactional é ignorado. Solução: extraia o método para outro @Service ou use self-injection.

Specification Pattern — Queries Dinâmicas

public class OrderSpecifications {

    public static Specification<Order> hasStatus(OrderStatus status) {
        return (root, query, cb) ->
            status == null ? null : cb.equal(root.get("status"), status);
    }

    public static Specification<Order> hasCustomer(Long customerId) {
        return (root, query, cb) ->
            customerId == null ? null : cb.equal(root.get("customerId"), customerId);
    }
}

💼 Perguntas de Entrevista — Fase 02

O que é o problema N+1 no JPA e como resolver?
N+1 ocorre quando buscar N entidades gera 1 + N queries (1 para o pai, N para filhos lazy). Soluções: JOIN FETCH (JPQL), @EntityGraph (Spring Data), @BatchSize (lote, safe com paginação).
O que é Optimistic Locking?
Estratégia que permite acesso concorrente sem bloquear registros. Usa @Version — se dois usuários lerem versão 1 e um salvar primeiro (versão → 2), o segundo recebe OptimisticLockException. Ideal para cenários de alta leitura e baixa escrita concorrente.
O que é a armadilha do self-invocation em @Transactional?
Chamar método @Transactional de outro método da mesma classe não ativa o proxy AOP do Spring. O @Transactional é ignorado. Solução: extrair método para outro @Service, usar self-injection, ou ApplicationContext.getBean().
Specification Pattern vs JPQL — quando usar qual?
Specification: queries dinâmicas com combinação de filtros opcionais (type-safe, reutilizável). JPQL: queries fixas e simples. Se o endpoint aceita 5+ filtros opcionais, Specification é mais limpo que montar JPQL com ifs.

JwtService — Geração e Validação de Tokens

@Service
public class JwtService {

    private final SecretKey key;
    private final long expirationMs;

    public JwtService(JwtProperties props) {
        this.key = Keys.hmacShaKeyFor(props.secretKey().getBytes());
        this.expirationMs = props.expirationMs();
    }

    public String generateToken(String userId, String roles) {
        return Jwts.builder()
            .subject(userId)
            .claim("roles", roles)
            .issuedAt(new Date())
            .expiration(new Date(System.currentTimeMillis() + expirationMs))
            .signWith(key)
            .compact();
    }

    public Claims parseToken(String token) {
        return Jwts.parser()
            .verifyWith(key)
            .build()
            .parseSignedClaims(token)
            .getPayload();
    }
}Java

💡 Por que desabilitar CSRF?

CSRF protege contra requisições forjadas em browsers com cookies de sessão. APIs stateless com JWT no header Authorization não usam cookies → CSRF não se aplica e pode ser desabilitado.

💼 Perguntas de Entrevista — Fase 03

Diferença entre JWT e sessão tradicional?
Sessão: estado no servidor (HttpSession), necessita sticky sessions ou Redis para múltiplas instâncias. JWT: stateless, token self-contained, qualquer instância valida. JWT escala horizontalmente sem infraestrutura compartilhada, mas não permite revogação individual sem blacklist.
Por que CSRF é desabilitado em APIs stateless?
CSRF explora cookies enviados automaticamente pelo browser. APIs JWT usam header Authorization (não cookie) → o browser não envia automaticamente → CSRF não se aplica.
O que é refresh token e por que usar?
Access token tem vida curta (15min-1h). Refresh token tem vida longa (dias/semanas) e é usado exclusivamente para gerar novos access tokens. Se o access token vazar, o impacto é limitado pelo TTL curto.
OAuth2 vs JWT: qual a relação?
OAuth2 é um protocolo de autorização (define fluxos como Authorization Code, Client Credentials). JWT é um formato de token. OAuth2 PODE usar JWT como formato de access token, mas não é obrigatório. São conceitos complementares, não concorrentes.

💼 Perguntas de Entrevista — Fase 04

O que é uma Dead Letter Queue (DLQ)?
Tópico especial que armazena mensagens que falharam após N retries. Permite investigar e reprocessar falhas sem perder mensagens. No Kafka: sufixo .DLT (Dead Letter Topic).
Kafka vs RabbitMQ — quando usar qual?
Kafka: alto throughput, log append-only, replay de mensagens, ideal para event streaming e event sourcing. RabbitMQ: message broker clássico, routing complexo, confirmação granular. Kafka para eventos em escala; RabbitMQ para task queues com routing sofisticado.
O que garante a ordem de mensagens no Kafka?
Ordem é garantida DENTRO de uma partition. Usando orderId como key, todos os eventos do mesmo pedido vão para a mesma partition → processados em ordem. Sem key, distribuição round-robin sem garantia de ordem.

        assertThat(response.getStatusCode()).isEqualTo(HttpStatus.CREATED);

💼 Perguntas de Entrevista — Fase 05

Diferença entre @Mock e @MockitoBean?
@Mock (Mockito puro) cria mock sem Spring Context — rápido, para testes unitários. @MockitoBean (Spring Boot 3.4+, substitui o deprecated @MockBean) substitui bean no Spring Context — lento, para testes de integração (@WebMvcTest, @SpringBootTest). Use @Mock sempre que possível.
O que é Testcontainers e por que usar?
Library que sobe containers Docker reais (PostgreSQL, Kafka, Redis) durante testes. Testa contra a mesma tecnologia de produção, eliminando H2 fakes. Lifecycle gerenciado automaticamente com @Testcontainers + @Container.
Como organizar testes na pirâmide?
Base (muitos): unitários de domínio com JUnit + AssertJ — testam regras de negócio sem framework. Meio: @WebMvcTest para controllers, @DataJpaTest para repositories. Topo (poucos): @SpringBootTest + Testcontainers para fluxos completos.

Dockerfile — Multi-Stage Build

FROM eclipse-temurin:21-jdk-alpine AS builder

RUN ./mvnw package -DskipTests

# Stage 2: Runtime com JRE (imagem final leve)

WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar

EXPOSE 8081
ENTRYPOINT ["java", "-jar", "app.jar"]Dockerfile
✅ Por que Multi-Stage?

JDK Alpine: ~340MB → JRE Alpine: ~180MB. Multi-stage descarta o JDK e mantém só o JRE na imagem final. Menos camadas + imagem menor = deploy mais rápido.

Health Checks

Health checks garantem que o Docker Compose espere serviços estarem prontos, não apenas rodando. O depends_on com service_healthy só inicia o serviço dependente quando o health check passar.

💼 Perguntas de Entrevista — Fase 07

O que é multi-stage build e por que usar?
Dockerfile com múltiplos FROM. Stage 1: build com JDK (pesado). Stage 2: runtime com JRE (leve). O build é descartado, resultando em imagem menor (40-50% menor), mais segura (sem ferramentas de build) e mais rápida para deploy.
Diferença entre CMD e ENTRYPOINT no Dockerfile?
ENTRYPOINT define o executável principal (não facilmente sobrescrito). CMD define argumentos default (facilmente sobrescrito pelo usuário). Combine: ENTRYPOINT ["java", "-jar"] e CMD ["app.jar"] para flexibilidade. Na prática, ENTRYPOINT é o comando fixo, CMD são parâmetros opcionais.
Por que depends_on sozinho não é suficiente?
depends_on só garante que o container INICIOU, não que está PRONTO. PostgreSQL pode demorar para aceitar conexões. Com healthcheck + condition: service_healthy, o Compose espera o banco responder antes de iniciar o app.

💡 Estados do Circuit Breaker

CLOSED (normal) → falhas excedem threshold → OPEN (rejeita chamadas) → tempo passa → HALF-OPEN (permite poucas chamadas de teste) → se OK volta a CLOSED, se falhas persistem volta a OPEN.

💼 Perguntas de Entrevista — Fase 08

O que é Service Discovery e por que usar?
Mecanismo onde serviços se registram (Eureka) e descobrem uns aos outros por nome lógico, sem hardcoded IPs/ports. Essencial em ambientes com auto-scaling onde instâncias sobem e descem dinamicamente.
O que é Circuit Breaker e quando usar?
Padrão que protege contra falhas em cascata. Se um serviço downstream está falhando, o circuit "abre" e retorna fallback imediato em vez de esperar timeout. Previne que um serviço lento derrube todos os que o chamam.
Diferença entre fallback e fail-fast?
Fallback: retorna resposta alternativa (cache, default, mensagem amigável). Fail-fast: lança exceção imediata. Fallback é melhor UX; fail-fast é mais honesto. Escolha por contexto: GET pode retornar cache; POST deve fail-fast.
O que é o padrão Saga em microserviços?
Padrão para transações distribuídas: sequência de transações locais, cada uma publica evento para a próxima. Se uma etapa falha, sagas compensatórias desfazem as anteriores. Dois tipos: Orquestração (coordenador central) e Coreografia (eventos diretos).

💼 Perguntas de Entrevista — Fase 09

Diferença entre CI e CD?
CI (Continuous Integration): automatizar build + testes a cada push — detecta erros cedo. CD pode ser Continuous Delivery (deploy automático até staging, manual para prod) ou Continuous Deployment (deploy automático até prod, sem aprovação).
O que é um quality gate?
Barreira automática no pipeline: se cobertura < 80% (JaCoCo), se SpotBugs encontrar bugs, se linting falhar — o pipeline BLOQUEIA. Impede código ruim de ir para produção.
Por que cache de dependências no CI?
Maven baixa dependências do repositório central a cada build. Cache (~/.m2/repository) persiste entre execuções → builds 50-70% mais rápidos. GitHub Actions: actions/cache com hash do pom.xml como key.

💼 Perguntas de Entrevista — Fase 10

Diferença entre logs, métricas e traces?
Logs: eventos discretos ("o que aconteceu"). Métricas: dados numéricos agregados ("qual o estado atual" — CPU, latência, throughput). Traces: caminho de um request entre serviços ("onde está o gargalo"). Os 3 são complementares — juntos dão observabilidade completa.
O que é Distributed Tracing e como funciona?
Rastreia um request através de múltiplos serviços: gera TraceId único no gateway, propaga via headers (B3/W3C) em cada chamada HTTP/Kafka. Cada serviço gera spans (unidades de trabalho). Zipkin/Jaeger visualiza o trace completo em cascata.
O que é o padrão RED?
Rate (requests/s), Errors (5xx/s), Duration (latência p95/p99). Métricas mínimas para monitorar qualquer serviço web. Se RED está saudável, o serviço está saudável. Alternativa: USE (Utilization, Saturation, Errors) para recursos infra.
Como correlacionar logs entre microserviços?
Micrometer Tracing injeta automaticamente traceId e spanId no MDC (Mapped Diagnostic Context). O logging pattern inclui %X{traceId}. Para correlacionar: grep traceId nos logs de todos os serviços → timeline completa do request.
